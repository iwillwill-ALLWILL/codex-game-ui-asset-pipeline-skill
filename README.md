# Game UI Asset Pipeline Skill

这个 Skill 用来做一件事：

把提示词、参考图、用户上传的 UI 截图，变成能放进游戏项目里的独立 UI 组件。

它适合做：

- 面板背景图
- 按钮和按钮状态
- 进度条
- HUD 框
- 背包格子
- 地图节点
- 图标
- 九宫格可拉伸 UI
- 一整套同画风 UI 组件包
- 安全背景色选择和边缘残留清理
- 从大到小的递进拆分
- 分层目录输出

它不是单独的生图模型。它会让当前平台里已有的识图、生图、抠图、透明背景、图集、游戏引擎接入能力一起工作，然后把结果整理成可用的游戏 UI 资产包。

最推荐的用法不是“从一张完整概念图里硬拆所有组件”，而是：

1. 先把用户主动上传的参考图、参考提示词、色卡、UI 截图、风格说明沉淀成项目风格库。
2. 再根据风格库直接生成全套常用 UI 组件，或生成用户当前指定的 UI 组件/素材。
3. 最后把生成好的透明 PNG 打包成 Godot 默认可用的组件包。

项目风格库不是一张静态参考图。它会随着用户持续上传资料、确认好结果、反馈坏结果而自组织：保存参考图片、参考提示词、成功提示词、失败避坑、色卡、材质规则、边框规则、图标规则和组件命名习惯。

概念图拆分仍然支持，但它更适合快速试拆、复用截图里已经清晰可见的元素、或做风格分析。概念图不是源文件分层，边框和背景粘在一起时，不可能保证每个边缘都像 PSD/Figma 那样完整干净。

## 安装方式

### 方式一：安装到 Codex

在 Codex 里复制这段话：

```text
请用 $skill-installer 从 GitHub 安装这个 Skill：

repo: iwillwill-ALLWILL/codex-game-ui-asset-pipeline-skill
path: skills/game-ui-asset-pipeline

安装完成后提醒我重启 Codex。
```

重启 Codex 后，就可以在对话里使用：

```text
使用 $game-ui-asset-pipeline ...
```

### 方式二：安装到其他有识图和生图能力的平台

如果你用的不是 Codex，而是其他支持 Agent、Skills、插件、工作流、工具说明或知识库的平台，复制这段话给那个平台里的 AI：

```text
请读取这个 GitHub 仓库：
https://github.com/iwillwill-ALLWILL/codex-game-ui-asset-pipeline-skill

请把 skills/game-ui-asset-pipeline 作为一个可复用的游戏 UI 资产生成 Skill 安装到当前平台。

要求：
1. 保留 SKILL.md 里的触发说明和工作流。
2. 保留 references 里的工具选择、组件命名、风格库规则。
3. 保留 scripts 里的分层打包、overview 概览图、Godot 输出、风格库入库能力。
4. 如果当前平台不能直接运行 Python 脚本，就把脚本能力改造成当前平台可用的工具节点或工作流。
5. 安装后告诉我应该用什么话术调用它。
```

这个 Skill 要求目标平台至少具备识图、生图或图片处理、文件处理能力；否则只能当提示词参考，不能完整产出资源。

| 能力 | 用来做什么 |
|---|---|
| 识图 | 分析参考图里的风格、色卡、组件结构 |
| 生图 | 根据提示词或参考图生成 UI 组件 |
| 图片编辑/抠图 | 把截图里的 UI 扣成透明 PNG |
| 文件读写 | 保存组件、overview 概览图、风格库 |
| 脚本/工作流 | 批量命名、检查、打包、输出引擎文件 |

如果平台只有聊天能力，没有识图、生图/图片处理、文件处理能力，就不要把它包装成“可直接产出 UI 组件”的 Skill。

## 它有哪几种用法

### 推荐使用顺序

| 优先级 | 用法 | 适合场景 |
|---|---|---|
| 1 | 项目风格库 + 全套常用 UI 组件 | 做正式游戏 UI 组件包，要求覆盖面大、风格统一、可复用 |
| 2 | 项目风格库 + 指定 UI 组件/素材 | 当前只缺某些按钮、面板、图标、道具素材 |
| 3 | 当前参考图 + 直接生成独立 UI 组件 | 临时根据一张图扩展同风格按钮、面板、进度条 |
| 4 | 从概念图或截图里拆组件 | 快速试拆、复用清晰可见元素、分析层级和风格 |
| 5 | 已有 PNG 打包 | 已经有透明素材，只需要整理成 Godot 可用结构 |

### 1. 首推：沉淀项目风格库

适合团队或课程反复做同一个游戏项目风格。你可以持续上传参考图、UI 截图、风格说明、参考提示词、成功生成提示词和反馈。

复制这段：

```text
使用 $game-ui-asset-pipeline，把我上传的资料沉淀成一个长期可复用的项目 UI 风格库。

风格名：<填写风格名>

要求：
1. 只保存我明确要求沉淀的资料。
2. 保存参考图片、参考提示词、风格说明和后续成功提示词。
3. 从参考图里提取色卡。
4. 总结线条、材质、边框、角标、发光、图标、按钮状态规则。
5. 形成风格卡、色卡和提示词库。
6. 后续生成新 UI 时，必须优先读取这个风格库。
7. 如果我反馈某次结果好，把它沉淀成更强参考；如果不好，把失败原因写进避坑规则。
```

风格库内部会维护这些内容：

| 内容 | 作用 |
|---|---|
| 参考图片 | 锁定真实画风、色彩、材质和构图语言 |
| 参考提示词 | 复用用户认可的描述方式 |
| 成功生成提示词 | 作为后续批量生成的高权重经验 |
| 失败反馈 | 形成 avoid-list，避免反复跑偏 |
| 色卡 | 控制整套 UI 的颜色一致性 |
| 风格卡 | 固定线条、边框、按钮、图标、材质规则 |
| 标签和索引 | 让多个项目风格库能被检索和区分 |

### 2. 生成全套常用 UI 组件

适合正式项目起步，直接覆盖大部分常见游戏 UI 需求。

复制这段：

```text
使用 $game-ui-asset-pipeline，读取 <风格名> 项目风格库，生成一套完整常用游戏 UI 组件包。

要求：
1. 不要生成完整游戏截图，只生成独立 UI 组件。
2. 按内置完整 UI 组件清单生成，覆盖尽可能多的通用需求。
3. 至少包含：面板、弹窗、按钮全状态、进度条、滑条、HUD、卡牌、背包格子、装备格、技能按钮、地图节点、商店/背包/任务/设置相关组件、图标、装饰件。
4. 不要把文字烘焙进图片，文字后面交给游戏引擎。
5. 所有组件画风、色卡、描边、材质保持一致。
6. 面板、按钮、卡框要适合九宫格拉伸。
7. 体量太大时按类别分批生成，但最后整理成一个组件包。
8. 输出目录按类别分类，比如 panels、buttons、bars、cards、slots、hud、icons、frames、images。
9. 默认只输出 Godot 文件，不要输出 Unity/Cocos/JSON/debug。
```

完整清单覆盖这些大类：

| 类别 | 例子 |
|---|---|
| foundations/panels | 大中小窗口、弹窗、tooltip、顶部栏、侧栏、分割线、角标、底纹 |
| buttons | primary、secondary、danger、confirm、cancel、tab、toggle、checkbox、radio，全状态 |
| bars | health、mana、stamina、xp、loading、cooldown、capacity、slider |
| cards/slots | 角色卡、物品卡、技能卡、任务卡、奖励卡、背包格、装备格、稀有度框 |
| hud | 血量资源区、货币区、任务追踪、通知、迷你地图、计时器、波次、分数 |
| menus | 主菜单、暂停、设置、背包、装备、技能、商店、制作、任务、结算 |
| map/progression | 地图节点、路径、当前/完成/锁定/商店/boss/事件状态 |
| battle/status | 技能按钮、回合顺序、目标标记、buff/debuff、伤害/治疗标签 |
| icons | 导航、资源、属性、状态、稀有度、职业、地图、操作图标 |
| decoration atoms | 铆钉、边角、徽章、横幅、丝带、发光、划痕、材质贴片 |

### 3. 根据风格库生成用户指定组件或素材

适合项目中途补素材。

```text
使用 $game-ui-asset-pipeline，读取 <风格名> 项目风格库，只生成我这次需要的 UI 组件。

目标组件：
- 4 个技能按钮状态
- 1 个 boss 血条底图和填充
- 8 个状态图标
- 1 个奖励弹窗面板
- 3 个稀有度物品框

要求：
1. 只生成这些组件，不要全套。
2. 仍然沿用风格库里的色卡、线条、材质和按钮状态规则。
3. 输出透明 PNG、overview.png 和 Godot 文件。
4. 按类别整理目录。
```

### 4. 次选：从用户上传的 UI 截图里直接扣组件

适合你已经有一张好看的 UI 截图，只想把里面清晰可见的组件拆出来。

注意：这不是最推荐的正式产线。截图里的 UI 边框、阴影、文字、背景场景常常已经烘焙在同一张图里，AI 只能从可见像素里判断边界。背景残留可以清理得比较好，但“框架边缘少一块或多一块”有时是源图边界不可分导致的。

```text
使用 $game-ui-asset-pipeline，从我上传的 UI 截图里扣出可复用组件。

要求：
1. 先根据这张图设计从大到小的拆分阶梯，不要机械套固定五层。
2. 默认输出所有有用层级。
3. 每一层单独一个文件夹。
4. 每个组件单独输出透明 PNG。
5. 检查边框是否少切、多切。
6. 如果源图边界不可分，明确说明，并优先建议按风格库重新生成该组件。
7. 默认打包成 Godot 可用的 UI 组件包。
```

拆分阶梯按从大到小理解，但层数要根据具体 UI 自适应：

| 示例层级 | 作用 | 例子 |
|---|---|---|
| `level_01_card_complete` | 完整对象不变，只做清理和透明化 | 完整卡牌、完整任务面板 |
| `level_02_card_frame_dividers_decor` | 外框 + 内部分割线 + 内部装饰 | 卡牌框架带装饰 |
| `level_03_card_frame_dividers` | 外框 + 内部分割线 | 卡牌框架和参考线 |
| `level_04_card_outer_frame` | 只保留外轮廓 | 卡牌外框 |
| `level_05_card_atomic_parts` | 彻底拆成独立小层 | 外框、分割线、装饰、背景、icon、人物立绘 |

这张卡牌适合五层，所以示例里是五层。简单按钮可能只需要三层，复杂 HUD 可能需要四层或七层。原则是：只要这个粒度对用户有用，就输出；没有独立价值的空层不要输出。不同粒度层级必须单独放文件夹；如果用户只要某一层，再按用户要求缩小范围。

### 5. 清理 UI 素材边缘的粉色颗粒

很多生图流程会用粉色背景做临时透明底。直接裁切会留下粉色小点。

复制这段：

```text
使用 $game-ui-asset-pipeline，清理这些 UI 素材周围的粉色边缘颗粒。

要求：
1. 这不是普通裁切问题，要按 chroma-key spill 处理。
2. 先判断当前 key color 是否和 UI 色卡撞色；如果撞色，换一个背景色或重新生成透明 PNG。
3. 如果是整张 atlas，先清理 atlas，再切组件。
4. 使用 soft matte、despill、边缘收缩和轻微羽化。
5. 清理后重新打包。
6. 检查 overview.png 和 PNG 边缘，不要再有背景残留或 key color 颗粒。
```

### 6. 处理概念图拆分里的边缘问题

如果不是粉色/绿色 key color 残留，而是“框架边缘少了一块、多了一块、跟背景粘在一起”，要按边界识别问题处理：

```text
使用 $game-ui-asset-pipeline，重新处理这批从概念图里拆出来的 UI 组件边缘。

要求：
1. 先判断问题是背景残留，还是 UI 边框和背景不可分。
2. 对每个组件先保守扩大裁切范围，不要把边框切掉。
3. 用现有开源分割/抠图/alpha matting 工具生成 mask，不要只靠颜色阈值。
4. mask 后处理时可以 close/dilate 1-3px 保住描边，再做轻微 feather。
5. 在深色和浅色背景上检查边缘，少切和多切都要返工。
6. 如果源图本身边缘被文字、阴影、场景背景遮挡，就不要继续硬拆，改成根据风格库重新生成这个组件。
```

可用的开源方向：

| 工具 | 适合做什么 |
|---|---|
| [SAM 2](https://github.com/facebookresearch/sam2) | 点选/框选式分割 UI 大块区域，先拿到粗 mask |
| [BiRefNet](https://github.com/ZhengPeng7/BiRefNet) | 高质量前景分割，适合主体和背景可分的 UI 元素 |
| [rembg](https://github.com/danielgatis/rembg) | 批量背景移除，适合简单独立素材 |
| [PyMatting](https://github.com/pymatting/pymatting) | 有 trimap 时做 alpha matting，处理抗锯齿、发光、软边 |

这些工具能明显减少边缘错误，但不能从一张烘焙概念图里恢复不存在的源图层。正式生产建议还是走“风格库生成独立组件”。

### 7. 把已有 PNG 打包成游戏 UI 组件包

适合你已经有一组透明 PNG，只需要整理成游戏项目能用的结构。

复制这段：

```text
使用 $game-ui-asset-pipeline，把我提供的 PNG 文件夹打包成游戏 UI 组件包。

要求：
1. 自动识别 panel、button、progress_bar、icon、slot。
2. 自动识别按钮状态 normal、hover、pressed、disabled。
3. 自动识别进度条 background 和 fill。
4. 每个层级单独一个文件夹。
5. 如果素材很多，`png/` 下面按 panels、buttons、bars、icons 等类别分文件夹。
6. 根目录保留一个总 `overview.png`。
7. 默认不要输出 JSON、debug 文件夹、中间裁剪图、三套引擎文件。
```

## 产出物是什么

一次完整任务通常会得到这些东西：

| 产出 | 作用 |
|---|---|
| `overview.png` | 总览所有层级 |
| `level_01_complete/png/` | 最大粒度 PNG，小包可以平铺 |
| `level_01_complete/png/buttons/` | 大包里的按钮分类 |
| `level_01_complete/png/icons/` | 大包里的图标分类 |
| `level_01_complete/godot/` | 这一层的 Godot 文件 |
| `level_02_structured/png/` | 更细一层的 PNG |
| `level_xxx/overview.png` | 每一层自己的概览图 |

组件包默认不输出 JSON、debug 文件夹、中间裁剪图或三套引擎文件。风格库内部可以维护色卡和风格卡，但不要混进组件交付目录。

## 这个 Skill 主要解决什么问题

| 问题 | 解决方式 |
|---|---|
| 每次生图都像不同游戏 | 用项目风格库锁定色卡、材质、线条、边框和提示词习惯 |
| 不知道该生成哪些 UI | 用完整组件清单覆盖常见游戏 UI 需求 |
| 截图好看但不能直接进游戏 | 把截图拆成独立透明组件 |
| UI 图片有背景残留或粉色边缘颗粒 | 先选择安全 key color，再用 chroma-key 清理流程处理边缘污染 |
| 按钮、进度条、面板命名混乱 | 按层级和组件名称整理 PNG |
| 素材给到引擎还要重新配置 | 默认输出 Godot 起步导入结构 |
| 不知道该切多细 | 根据当前 UI 自适应设计从大到小的拆分阶梯，默认输出全部有用层级 |
| 课程或团队想复用同一画风 | 把用户主动上传的资料沉淀进本地风格库 |

## 使用时要告诉 AI 的关键信息

你不需要讲技术细节，只要把这几件事说清楚：

```text
我要做什么游戏画风？
我上传的图是参考风格，还是要直接扣组件？
我要全套常用 UI，还是只要指定组件？
有没有参考提示词或成功提示词要一起沉淀？
是否只要某一层？如果没有特别说明，默认输出全部有用层级。
要不要按钮状态？
要不要进度条 background / fill？
目标引擎是否就是默认 Godot？
是否要把这次资料沉淀成长期风格库？
```
